# Robustness of German Offensive Language Detection

This project was created for the course *Python for NLP*. It investigates how well German offensive-language classifiers handle the kind of noisy writing that is common on social media.

The main research question is:

> How robust is a German offensive-language classifier to surface-level noise such as typos, unusual casing, informal spelling, word elongation, and slang? Can differences in tokenization help explain why the models become less accurate?

Two different models are compared:

* a TF-IDF model with Logistic Regression
* a fine-tuned `deepset/gbert-base` transformer

Both models are trained and evaluated on the GermEval 2018 offensive-language dataset. Their performance is measured using macro-F1 on the original clean test set and on several controlled, perturbed versions of the same test data.

## Environment

The project uses Python 3.13, as specified in `.python-version`. Dependencies are managed with [uv](https://docs.astral.sh/uv/) through `pyproject.toml` and `uv.lock`.

To create the environment with the exact dependency versions used in the project, run:

```bash
uv sync
```

This creates a local `.venv/` environment.

The notebooks should be run with the kernel from this environment, for example:

```bash
.venv/bin/jupyter lab
```

The main dependencies are:

* `pandas`
* `scikit-learn`
* `torch`
* `transformers`
* `sentencepiece`

The `transformers` package is pinned to a version below 5. Version 5 removed support for loading the older `vocab.txt`-only tokenizer format used by `gbert-base` when no `tokenizer.json` is available.

Notebook 05, which fine-tunes the transformer, is designed to run on Google Colab using a free T4 GPU. The transformer was therefore not trained locally.

All other notebooks were run locally on a MacBook with Apple Silicon. For transformer inference in notebook 06, PyTorch's MPS backend is used where available.

## Data

The project uses the **GermEval 2018 Shared Task on the Identification of Offensive Language** dataset:

https://github.com/uds-lsv/GermEval-2018-Data

The dataset is licensed under **CC BY 4.0**.

It contains:

* 5,009 training tweets
* 3,532 test tweets
* approximately 34% offensive examples

The main binary labels are:

* `OFFENSE`
* `OTHER`

The dataset is described in the following publication:

> Michael Wiegand, Melanie Siegel, and Josef Ruppenhofer. “Overview of the GermEval 2018 Shared Task on the Identification of Offensive Language.” Proceedings of GermEval, 2018, Vienna, Austria.

To use the data, download or clone the original repository and place its contents in:

```text
data/raw/GermEval-2018-Data-master/
```

This is the path expected by `src/config.py`.

Only the following two files are read by the project:

```text
germeval2018.training.txt
germeval2018.test.txt
```

The raw data, processed train/validation/test splits, and generated perturbation files are excluded from version control through `.gitignore`. They can all be regenerated by running notebooks 02 and 03.

## Reproducing the experiments

The notebooks should be run in the following order.

### 1. `01_data_exploration.ipynb`

This notebook performs an initial exploration of the raw GermEval data.

It looks at aspects such as:

* class balance
* text length
* the general structure of the dataset

The notebook is only used for exploration. Its outputs are not required by the later steps.

### 2. `02_preprocessing_and_split.ipynb`

This notebook performs the main preprocessing.

The text is cleaned by:

* replacing user mentions with a placeholder
* normalizing whitespace
* removing or normalizing line breaks

The official training file is then split into training and validation sets.

The split is stratified by the binary label, uses a fixed random seed of 42, and assigns 15% of the original training data to the validation set.

The official GermEval test set remains unchanged and is kept separate throughout model development.

The notebook creates:

```text
data/processed/train.csv
data/processed/val.csv
data/processed/test.csv
```

### 3. `03_perturbations.ipynb`

This notebook creates controlled noisy versions of the official test set.

Five types of perturbations are used:

* typos
* casing variation
* umlaut replacement
* word elongation
* slang replacement

Each perturbation is applied at three intensity levels:

* 5%
* 10%
* 20%

The perturbations are intended to preserve the original label while changing only the surface form of the text.

A manual inspection is also included to check that the generated examples still have the same meaning and label.

The resulting files are saved in:

```text
data/perturbed/
```

### 4. `04_baseline_model.ipynb`

This notebook trains the TF-IDF and Logistic Regression baseline.

A grid search is performed over:

* unigram and unigram-plus-bigram TF-IDF features
* different values of the Logistic Regression parameter `C`

The models are compared using macro-F1 on the validation set.

The best-performing vectorizer and classifier are saved in:

```text
models/
```

Validation metrics and the grid-search results are saved in:

```text
results/
```

### 5. `05_transformer_model.ipynb`

This notebook fine-tunes `deepset/gbert-base`.

The model is trained three times using the seeds:

```text
42, 123, 67
```

This notebook is intended to run on Google Colab.

To run it, upload the following files and folders to Google Drive:

```text
05_transformer_model.ipynb
src/
data/processed/train.csv
data/processed/val.csv
```

The setup cell mounts Google Drive and installs the required dependencies.

For each seed, the notebook saves the best checkpoint based on validation macro-F1, together with the training history.

The outputs are stored in:

```text
results/transformer/gbert-base/seed_<seed>/
```

### 6. `06_robustness_evaluation.ipynb`

This notebook performs the final evaluation.

It loads both trained models and evaluates them on:

* the clean official test set
* every perturbed version of the test set

This is the only stage of the project where the official test set is used for the actual final model evaluation.

In addition to measuring macro-F1, the notebook investigates why the models degrade under different perturbations.

For the transformer, it measures the **WordPiece fragmentation ratio**. This describes how much more strongly a perturbed text is split into subword tokens compared with the clean version.

For the TF-IDF baseline, it measures the **out-of-vocabulary rate**, or OOV rate. This is the share of words in a perturbed text that are not present in the vocabulary learned by the TF-IDF vectorizer.

The notebook runs locally because it only performs inference.

All final result tables are saved in:

```text
results/robustness/
```

## Seeds and hyperparameters

The main seeds and hyperparameters are defined centrally in `src/config.py` and imported by the notebooks. This avoids defining the same settings in several different places.

| Setting                              | Value                    |
| ------------------------------------ | ------------------------ |
| General seed for splits and baseline | `SEED = 42`              |
| Validation split size                | `VAL_SIZE = 0.15`        |
| Perturbation intensities             | `[0.05, 0.10, 0.20]`     |
| Baseline n-gram ranges               | `[(1,1), (1,2)]`         |
| Baseline `C` values                  | `[0.01, 0.1, 1.0, 10.0]` |
| Transformer checkpoint               | `deepset/gbert-base`     |
| Transformer training seeds           | `[42, 123, 67]`          |
| Learning rate                        | `2e-5`                   |
| Batch size                           | `16`                     |
| Number of epochs                     | `4`                      |
| Weight decay                         | `0.01`                   |
| Maximum sequence length              | `128`                    |

During transformer training, the checkpoint with the highest validation macro-F1 is kept.

The maximum sequence length of 128 was selected based on the token-length analysis in notebook 05. It covers approximately 99.9% of the tokenized training texts without truncation.

## Results

The main result files are:

```text
results/baseline_grid_search.csv
results/baseline_val_report.txt
```

These contain the baseline hyperparameter search and validation classification report from notebook 04.

For the transformer, each seed directory contains:

```text
results/transformer/gbert-base/seed_<seed>/training_history.csv
results/transformer/gbert-base/seed_<seed>/summary.csv
```

These files contain the per-epoch training history and the validation results across the three seeds.

The final robustness results are stored in:

```text
results/robustness/
```

This folder contains:

* clean and perturbed macro-F1 scores
* degradation values
* WordPiece fragmentation statistics
* TF-IDF OOV statistics
* `clean_reference.txt`

### Main findings

On the clean official test set, the baseline reaches a macro-F1 of:

```text
0.6445
```

The fine-tuned `gbert-base` model reaches:

```text
0.7731 ± 0.0039
```

The transformer therefore performs clearly better on the clean data.

However, both models lose performance when noise is introduced.

Typos cause the strongest overall degradation for both models.

Casing variation is mainly a weakness of the transformer. `gbert-base` uses a cased tokenizer, which means that changes in capitalization can affect its tokenization and representation.

The TF-IDF baseline is almost unaffected by casing changes because its vectorizer converts text to lowercase.

The relationship between tokenization changes and model degradation is also strong:

* the WordPiece fragmentation ratio has a correlation of `r = 0.92` with transformer degradation
* the TF-IDF OOV rate has a correlation of `r = 0.96` with baseline degradation

These results suggest that the models do not only differ in how accurate they are, but also in how and why they fail when the input text becomes noisy.
